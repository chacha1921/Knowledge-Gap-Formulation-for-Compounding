Slide 3 of 8 — KV Cache: What It Is & How It Works Source: Kwon et al., PagedAttention — SOSP 2023

What is KV Cache?

When a transformer processes tokens, each token generates a Key (K) and a Value (V) across every attention layer. These are stored in GPU memory so the model can attend to prior tokens without recomputing them. The stored collection of all K and V vectors is called the KV cache.

The Two Inference Phases

Phase 1 — Prefill

Building the cache

  • All input tokens processed in parallel — one forward pass
  • KV cache built and stored in GPU memory
  • Output: not a word — it is the KV cache itself
Phase 2 — Decode

Reading the cache

  • Tokens generated one at a time — cannot parallelize
  • Each step reads the full KV cache from GPU memory
  • Token 5 cannot start until tokens 1–4 are done
Within a single API call
KV cache IS reused
Decode reads the cache prefill built — no recomputation within one call
Across separate API calls
KV cache is DISCARDED
Every new turn = new request = full prefill recomputed from scratch

KV cache is NOT preserved across API calls

Every turn is a separate API request. My Conversion Engine re-pays full prefill cost on the full accumulated context — system prompt + all prior turns — every single turn.

Slide 6 of 8 — Problems This Gap Created in My Week 10 Work Conversion Engine · Tenacious ICP + bench summary + hiring signal brief re-sent every turn
⚠

No turn budget enforced

Without knowing cost is O(n²), I set no max_turns guard. task_4 ran uncapped turns and cost 14.7× more than task_73 — both received the same reward.

⚠

Prefix caching never applied

The Conversion Engine prepends a dynamic hiring signal brief (different per company) before the static system prompt — this means the byte-identical prefix requirement is never met. Every turn pays full prefill on the entire context.

⚠

Client cost estimate is wrong

Quoting "$0.0199 per conversation" to Tenacious would be incorrect. The actual range is $0.0068–$0.0998. High-turn edge cases cost 5× the quoted average.

⚠

p95 latency SLA is unmeasurable

p95 = 551s is reported as if it reflects inference time. It doesn't — task_92 (687s, $0.0124) proves the agent was waiting, not generating. Inference latency and environment wait are mixed and cannot be optimized separately.

⚠

Context window risk unmonitored

As turns accumulate, the growing context approaches the model's context limit. Without a turn or token budget, a long-running task could hit the limit mid-conversation and fail silently.

⚠

No cost cap per task

Nothing in the Conversion Engine stops a single task from consuming $0.10+. Without a hard max_cost guard, unpredictable tasks can blow the per-run budget without warning.

Slide 7 of 8 — Local Caching: What Can Be Done Three concrete fixes for the Conversion Engine
Fix 1 — Anthropic Prompt Caching (cache_control)

Pin the static system prompt as a cached prefix

# Restructure the prompt: static content first, dynamic last
response = anthropic.messages.create(
    model="claude-sonnet-4-6",
    system=[
        {"type": "text", "text": STATIC_SYSTEM_PROMPT,
         "cache_control": {"type": "ephemeral"}},  # ← cached prefix
        {"type": "text", "text": dynamic_hiring_brief}    # ← not cached, changes per company
    ],
    messages=conversation_history
)

Requirement: static content must be ≥1,024 tokens and appear before all dynamic content from position 0.

Fix 2 — Turn Budget + Context Pruning

Cap O(n²) growth at the source

# Hard cap on turns
MAX_TURNS = 5
if turn_count >= MAX_TURNS:
    raise TurnBudgetExceeded

# Keep only the last N turns to prune context
MAX_HISTORY = 6  # 3 turns × 2 messages
messages = messages[-MAX_HISTORY:]
Fix 3 — Cost Guard Per Task

Hard stop before context explodes

# Track cumulative cost per task run
MAX_COST_PER_TASK = 0.05  # $0.05 hard cap
cumulative_cost += (
    prompt_tokens * 3.00 / 1_000_000 +
    output_tokens * 15.00 / 1_000_000
)
if cumulative_cost > MAX_COST_PER_TASK:
    raise CostBudgetExceeded
